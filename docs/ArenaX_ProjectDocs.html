Design Principles

Security-First

Admin role verified against DB on every request — never trusted from JWT payload. Auth cache invalidated instantly on ban/delete.

Soft-Delete Everywhere

No hard deletes on content or accounts. Triggers snapshot to archive tables. Restore capability for admins. Audit log for compliance.

Validation at Every Layer

express-validator on all routes. Ownership checks in controllers. Input sanitization before DB write. No raw user content reaches DB unsanitized.

Real-time Where It Matters

Socket.IO for DMs, stream chat, tournament brackets, matchmaking queue, and presence — not for everything. REST for all non-realtime operations.

Performance Conscious

60s in-memory auth cache reduces DB hit ~95%. Matchmaking uses 3 parallel ELO queries instead of sequential. Connection pool with keepAlive.

Production Ready

Graceful shutdown handler (SIGTERM/SIGINT). Required ENV guard at startup. Retry-with-backoff DB connect. PM2 + Nginx + DEPLOY.md all prepared.

Database Design Principles

  • InnoDB engine on all tables — foreign keys, transactions, row-level locking
  • utf8mb4_unicode_ci — supports full emoji and Unicode (required for esports usernames)
  • Soft-delete pattern — status columns (active/banned/deleted) + archive tables via triggers
  • Reserved word quoting — rank, role backtick-quoted everywhere (MySQL 8 reserved)
  • Composite indexes on all FK pairs and common filter combos (status+deadline, user+game)
  • JSON columns for screenshots array (games table) — avoids a separate screenshots table
  • Archive system — 6 archive tables + archive_config + audit_log + stored procedure for purge
  • Constraint naming — all FK and check constraints explicitly named (fk_*, chk_*) for clarity

Socket.IO Architecture

Real-time layer — auth flow, rooms, events, and rate limiting.

Connection & Auth Flow

Client: socket.io-client
↓ handshake.auth.token = JWT
io.use() middleware
↓ verifyToken(token) — JWT decode
DB check: status = 'active'
↓ attach socket.user = decoded
socket.join(user:userId)
↓ all events now available
socket.emit('error') if banned/expired

Rooms

RoomJoined byPurpose
user:{id}Auto on connectPersonal DM inbox
stream:{id}join_stream eventStream chat + viewer count
tournament:{id}join_tournament eventBracket updates
queue:{gameId}join_queue eventMatchmaking queue

Rate Limiting

60 events / 60 seconds per userId
In-memory Map with sliding window
Stale entries pruned every 5 minutes
Applies to: send_message, stream_chat

Events Reference

📨 Direct Messages

CLIENT send_message Send DM — body: { receiverId, content }
SERVER new_message Emitted to receiver's room: user:{receiverId}
SERVER message_sent Confirmation back to sender

📺 Streams

CLIENT join_stream { streamId } — joins stream:{id} room
SERVER viewer_joined / viewer_left Broadcast to stream room
CLIENT stream_chat { streamId, message } — max 500 chars, rate limited
SERVER stream_chat_message { userId, username, message, timestamp } to stream room

🏆 Tournaments

CLIENT join_tournament { tournamentId } — joins tournament:{id} room
CLIENT match_update { tournamentId, matchData } — admin only, DB-verified
SERVER bracket_updated matchData broadcast to all in tournament room

🎮 Matchmaking & Presence

CLIENT join_queue / leave_queue { gameId } — join/leave queue:{gameId} room
CLIENT go_online Broadcasts user_online to all connected sockets
SERVER user_online / user_offline Broadcast on go_online / disconnect
SERVER error { message } — rate limit exceeded, invalid input, unauthorized

Services

matchmakingService.js

ELO-based player matching.

  • K=32 Elo calculation
  • 3 window sizes: ±200, ±400, ±600
  • All 3 queries run in parallel (Promise.all)
  • Returns narrowest match found

tournamentService.js

Bracket generation logic.

  • Single elimination bracket
  • Team seeding from registrations
  • Match creation for each round

gameDataService.js

Seeds games from JSON.

  • Reads backend/data/games.json
  • Upserts to games table
  • Called via POST /api/games/sync

Auth Cache System (db.js)

// In-memory Map — {userId → {expiresAt}}
isAuthCacheHit(userId)   → boolean   // TTL check
setAuthCache(userId)                 // prime cache (60s TTL)
invalidateAuthCache(userId)          // instant eviction on ban/delete

// Auto-sweep every 5 minutes (prevents memory leak on long-lived server)
setInterval(() => { /* prune expired entries */ }, 5 * 60_000)

On a typical request the cache hit rate is ~95% — the DB is only queried on the first request per user per 60-second window, or after a ban/delete event.
